# 情報セキュリティポリシー & コンプライアンス基準 (ISMS 準拠)

## 1. アクセス制御方針

### 1.1. 認証 (Authentication)

- 全従業員は **多要素認証 (MFA)** を有効化しなければならない。
- パスワードは以下の要件を満たすこと:
  - 12 文字以上
  - 英大文字・小文字・数字・記号の混在
  - 辞書攻撃に弱い単語の禁止
- 90 日ごとのパスワード変更は**廃止**する（NIST SP 800-63B 推奨準拠）。代わりに、漏洩検知時に即時リセットを行う。

### 1.2. 認可 (Authorization)

- **RBAC (Role-Based Access Control)** を原則とする。
- 特権 ID（Admin 権限）の利用は、申請ベースの一時的な昇格（JIT: Just-In-Time Access）のみ許可する。
- 退職者のアカウントは、退職日当日の 18:00 までに無効化（Disable）する。

## 2. データ保護

### 2.1. 暗号化

- **通信の暗号化**: すべての通信は TLS 1.2 以上を使用する。HTTP (80) は HTTPS (443) へ強制リダイレクトする。
- **保存データの暗号化**:
  - データベース: AWS KMS (Key Management Service) を用いた透過的暗号化 (TDE) を有効にする。
  - バックアップ: S3 バケットのサーバーサイド暗号化 (SSE-S3 or SSE-KMS) を強制する。

### 2.2. 個人情報 (PII) の取り扱い

- マイナンバー、クレジットカード情報などの機微情報は、原則として本システム内に保存しない。
- 必要な場合は、専用のトークン化サービス（PCI-DSS 準拠）を利用し、トークンのみを保存する。

## 3. 開発セキュリティ

### 3.1. セキュアコーディング

- **OWASP Top 10** の脆弱性（SQL インジェクション、XSS 等）を作り込まないよう、フレームワークの標準機能を利用する。
- 生の SQL クエリは原則禁止とし、ORM (SQLAlchemy 等) を使用する。

### 3.2. 脆弱性管理

- **SCA (Software Composition Analysis)**: GitHub Dependabot を有効化し、依存ライブラリの脆弱性を常時監視する。Critical な脆弱性は 24 時間以内にパッチを適用する。
- **SAST (Static Application Security Testing)**: CI パイプラインで SonarQube / CodeQL を実行し、マージをブロックする。

## 4. 監査ログ

- 以下の操作は必ず監査ログとして記録し、3 年間保存する（改ざん防止措置を含む）。
  - ログイン/ログアウト（成功・失敗問わず）
  - 個人データの閲覧・更新・削除
  - 権限変更操作
